重新排序增强型搜索在矢量检索后添加了第二个评分通道，因此 Mem0 可以首先返回最相关的记忆。当仅关键字相似性错过了细微差别时，或者当您需要最高置信度的上下文来进行代理决策时，请启用它。

<信息>
  **你会在以下情况下使用它......**
  - 查询是细致入微的，需要超出向量距离的语义理解。
  - 大型内存集合会产生太多接近的匹配项，无法手动查看。
  - 您希望通过将排名委托给专用模型来获得跨提供商的一致评分。
</信息>

<警告>
  重新排名会增加延迟，对于托管模型而言，还会增加 API 支出。对生产流量进行基准测试，并为延迟敏感的请求定义回退路径。</警告>

<注意>
  所有配置片段都直接转换为 TypeScript SDK——交换对象的字典，同时保留相同的键（“provider”、“config”、“rerank”标志）。
</注>

## 特征剖析- **初始向量搜索：** 通过相似性检索候选记忆。
- **Reranker pass：** 专门的模型根据原始查询对每个候选者进行评分。
- **重新排序的结果：** Mem0 在返回响应之前使用重新排序器的分数对响应进行排序。
- **可选的后备措施：** 如果性能或成本成为问题，则切换每个请求的重新排名或完全禁用它。<手风琴组>
  <Accordion title="支持的提供商">
    - **[Cohere](/components/rerankers/models/cohere)** – 具有基于 API 评分的多语言托管重新排序器。  
    - **[Sentence Transformer](/components/rerankers/models/sentence_transformer)** – 用于 GPU 或 CPU 的本地 Hugging Face 交叉编码器。  
    - **[Hugging Face](/components/rerankers/models/huggingface)** – 携带任何托管或本地重新排序模型 ID。  
    - **[LLM Reranker](/components/rerankers/models/llm_reranker)** – 使用您首选的 LLM（OpenAI 等）进行提示驱动评分。  
    - **[零熵](/components/rerankers/models/zero_entropy)** – 针对检索任务调整的高质量神经重新排序。</手风琴>

## 配置它

### 基本设置```python
from mem0 import Memory

config = {
    "reranker": {
        "provider": "cohere",
        "config": {
            "model": "rerank-english-v3.0",
            "api_key": "your-cohere-api-key"
        }
    }
}

m = Memory.from_config(config)
```
<信息图标=“检查”>
  确认 `results["results"][0]["score"]` 反映了重排序器输出 - 如果该字段丢失，则不会应用重排序器。
</信息>

<提示>
  将“top_k”设置为仍捕获相关命中的最小候选池。较小的池不断降低重新排名成本。
</提示>

### 特定于提供商的选项```python
# Cohere reranker
config = {
    "reranker": {
        "provider": "cohere",
        "config": {
            "model": "rerank-english-v3.0",
            "api_key": "your-cohere-api-key",
            "top_k": 10,
            "return_documents": True
        }
    }
}

# Sentence Transformer reranker
config = {
    "reranker": {
        "provider": "sentence_transformer",
        "config": {
            "model": "cross-encoder/ms-marco-MiniLM-L-6-v2",
            "device": "cuda",
            "max_length": 512
        }
    }
}

# Hugging Face reranker
config = {
    "reranker": {
        "provider": "huggingface",
        "config": {
            "model": "BAAI/bge-reranker-base",
            "device": "cuda",
            "batch_size": 32
        }
    }
}

# LLM-based reranker
config = {
    "reranker": {
        "provider": "llm_reranker",
        "config": {
            "provider": "openai",
            "model": "gpt-4o-mini",
            "api_key": "your-openai-api-key",
            "top_k": 5
        }
    }
}
```
<注意>
  当您将这些配置插入生产项目时，将身份验证密钥保留在环境变量中。
</注>

### 异步支持```python
from mem0 import AsyncMemory

async_memory = AsyncMemory.from_config(config)

async def search_with_rerank():
    return await async_memory.search(
        "What are my preferences?",
        filters={"user_id": "alice"},
        rerank=True
    )

import asyncio
results = asyncio.run(search_with_rerank())
```
<信息图标=“检查”>
  检查异步响应以确认重新排名仍然适用；分数应与同步实现相匹配。
</信息>

### 调整性能和成本```python
# GPU-friendly local reranker configuration
config = {
    "reranker": {
        "provider": "sentence_transformer",
        "config": {
            "model": "cross-encoder/ms-marco-MiniLM-L-6-v2",
            "device": "cuda",
            "batch_size": 32,
            "top_k": 10,
            "max_length": 256
        }
    }
}

# Smart toggle for hosted rerankers
def smart_search(query, user_id, use_rerank=None):
    if use_rerank is None:
        use_rerank = len(query.split()) > 3
    return m.search(query, filters={"user_id": user_id}, rerank=use_rerank)
```
<提示>
  使用启发式方法（查询长度、用户级别）来决定何时重新排名，以便高信号查询受益，而无需对每个请求进行征税。
</提示>

### 优雅地处理失败```python
try:
    results = m.search("test query", filters={"user_id": "alice"}, rerank=True)
except Exception as exc:
    print(f"Reranking failed: {exc}")
    results = m.search("test query", filters={"user_id": "alice"}, rerank=False)
```
<警告>
  始终回退到仅向量搜索——与稍微不太相关的排序相比，删除的查询会带来更大的准确性问题。
</警告>

### 从 v0.x 迁移```python
# Before: basic vector search
results = m.search("query", filters={"user_id": "alice"})

# After: same API with reranking enabled via config
config = {
    "reranker": {
        "provider": "sentence_transformer",
        "config": {
            "model": "cross-encoder/ms-marco-MiniLM-L-6-v2"
        }
    }
}

m = Memory.from_config(config)
results = m.search("query", filters={"user_id": "alice"})
```

## 查看实际效果

### 基本重排搜索```python
results = m.search(
    "What are my food preferences?",
    filters={"user_id": "alice"}
)

for result in results["results"]:
    print(f"Memory: {result['memory']}")
    print(f"Score: {result['score']}")
```
<信息图标=“检查”>
  希望每个结果都会列出重新排序器调整的分数，以便您可以将排序与基线向量结果进行比较。
</信息>

### 根据请求切换重新排名```python
results_with_rerank = m.search(
    "What movies do I like?",
    filters={"user_id": "alice"},
    rerank=True
)

results_without_rerank = m.search(
    "What movies do I like?",
    filters={"user_id": "alice"},
    rerank=False
)
```
<提示>
  在推出期间记录重新排名与未重新排名的列表，以便利益相关者可以在到处实施之前看到改进。
</提示>

<信息图标=“检查”>
  您应该在两个列表中看到相同的记忆，但重新排序的响应将根据语义相关性对它们重新排序。
</信息>

### 与元数据过滤器结合```python
results = m.search(
    "important work tasks",
    filters={
        "AND": [
            {"user_id": "alice"},
            {"category": "work"},
            {"priority": {"gte": 7}}
        ]
    },
    rerank=True,
    top_k=20
)
```
<信息图标=“检查”>
  验证过滤后的重新排名搜索仍然尊重每个元数据子句 - 重新排名仅对候选者重新排序，它永远不会绕过过滤器。
</信息>

## 验证该功能是否正常工作

- 检查“分数”（向量）和重新排序分数的结果有效负载；不匹配的字段表示重新排序器未执行。
- 跟踪启用重新排名之前和之后的延迟，以确保 SLA 有效。
- 查看提供商日志或仪表板以了解限制或配额警告。
- 在默认重新排名响应之前，运行 A/B 比较（重新排名开/关）以验证相关性的改善。

## 最佳实践1. **从本地开始：** 在支付托管 API 费用之前尝试 Sentence Transformer 模型以证明其价值。
2. **监控延迟：** 添加有关重新排序持续时间的指标，以便您快速注意到回归。
3. **控制支出：** 使用“top_k”和选择性切换来限制托管重新排序成本。
4. **保留后备：** 始终捕获重新排序器失败并继续仅向量排序。
5. **经常进行实验：** 交换提供商或模型以找到最适合您的领域和语言组合的方案。

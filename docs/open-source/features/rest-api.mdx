Mem0 REST API 服务器通过 HTTP 公开每个 OSS 内存操作。与堆栈一起运行它，可以从任何使用 REST 的语言中添加、搜索、更新和删除内存。

<信息>
  **你会在以下情况下使用它......**
  - 您的服务已经与 REST API 通信，并且您希望 Mem0 匹配该风格。
  - 使用没有 Mem0 SDK 的语言的团队仍然需要访问内存。
  - 您计划通过“/docs”中的内置 OpenAPI 页面探索或调试端点。
</信息>

<警告>
  **首次自托管，或从 1.x 之前的版本升级？** 从 [自托管设置](/open-source/setup) 开始。它遍历堆栈、设置向导以及依赖于开放端点或“ADMIN_API_KEY”的部署的升级路径。本页面仅涵盖 API 表面和身份验证模式。
</警告>

<警告>
  **OSS 与平台 API 路径：** 自托管 OSS 服务器**不**使用 `/v1/` 前缀。例如，端点是“POST /memories”，而不是“POST /v1/memories/”。 [API 参考](/api-reference) 在“api.mem0.ai”处记录了使用“/v1/”路径的托管平台——这些路径不适用于 OSS 服务器。
</警告><警告>
  在将服务器暴露给内部网络之外的任何内容之前，启用 API 密钥身份验证（见下文）和 HTTPS。
</警告>

## 特点

- **CRUD 端点：** 通过 `user_id`、`agent_id` 或 `run_id` 创建、检索、搜索、更新、删除和重置内存。
- **身份验证：** 默认情况下打开。仪表板会话使用 JWT；编程客户端使用每个用户的“X-API-Key”标头。仍支持旧版“ADMIN_API_KEY”。
- **状态健康检查：** 访问基本路由以确认服务器在线。
- **OpenAPI 浏览器：** 访问“/docs”进行交互式测试和架构参考。

## 身份验证

身份验证默认处于开启状态。受保护的端点需要 JWT（来自仪表板登录流程）或“X-API-Key”标头。 `/` 重定向、`/docs` 和 `/openapi.json` 路由保持打开状态，以便您可以访问 OpenAPI 资源管理器。|模式|如何发送 |何时使用它 |
|---|---|---|
|不记名 JWT | `授权：持有者 <access_token>` |仪表板会话；令牌来自“POST /auth/login”并通过“POST /auth/refresh”刷新 |
|每用户 API 密钥 | `X-API-密钥：m0sk_...` |编程访问范围仅限于单个仪表板用户 |
|旧版“ADMIN_API_KEY” | `X-API-Key：<环境值>` |设置“ADMIN_API_KEY”环境变量的部署的向后兼容
| `AUTH_DISABLED=true` | — |仅本地开发；完全绕过身份验证 |

`/docs` OpenAPI 浏览器支持两种身份验证模式。单击页面顶部的 **Authorize** 并粘贴“Bearer <access_token>” (JWT) 或您的“X-API-Key”值。受保护的端点将返回“401”，直到您授权为止。

### 登录并使用 JWT

注册第一个管理员（仅在尚不存在用户时有效），然后登录：```bash
# First admin only — returns 403 after the first admin is registered
curl -X POST http://localhost:8888/auth/register \
  -H "Content-Type: application/json" \
  -d '{"name": "Admin", "email": "admin@example.com", "password": "strong-password"}'
```

```bash
curl -X POST http://localhost:8888/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email": "admin@example.com", "password": "your-password"}'
```
使用返回的“access_token”作为不记名令牌：```bash
curl -X POST http://localhost:8888/memories \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{
    "messages": [{"role": "user", "content": "I love pizza."}],
    "user_id": "alice"
  }'
```
当访问令牌过期时，请在“POST /auth/refresh”处交换刷新令牌。

### 创建并使用每用户 API 密钥

从仪表板 **API Keys** 页面创建密钥，或使用 JWT 调用“POST /api-keys”。完整的“m0sk_...”值在创建时返回**一次** - 安全地存储它。```bash
curl -X POST http://localhost:8888/memories \
  -H "Content-Type: application/json" \
  -H "X-API-Key: m0sk_your_key_here" \
  -d '{
    "messages": [{"role": "user", "content": "I love pizza."}],
    "user_id": "alice"
  }'
```
每用户密钥继承创建用户的范围。通过“GET /api-keys”和“DELETE /api-keys/{id}”列出或撤销它们。

### 旧版`ADMIN_API_KEY`

设置“ADMIN_API_KEY”环境变量并将其作为“X-API-Key”发送。该请求被视为管理员级别，并且不与仪表板用户绑定。保留此模式是为了与旧的自托管部署向后兼容 - 对于新设置更喜欢 JWT 或每用户密钥。```bash
ADMIN_API_KEY=your-long-admin-key
```
<警告>
  设置“AUTH_DISABLED=true”会使每个受保护的端点打开——启用后，服务器会在启动时记录一条警告。当“ADMIN_API_KEY”短于 16 个字符时，服务器还会发出警告。切勿在生产中启用“AUTH_DISABLED”，并且如果您依赖旧版回退，请始终使用长“ADMIN_API_KEY”。
</警告>

## 端点参考

OSS REST 服务器公开以下端点。没有一个使用“/v1/”前缀。

### 内存操作|方法|路径|描述 |
|--------|------|-------------|
| `发布` | `/配置` |设置内存配置。使用 400 拒绝非捆绑提供商 |
| `获取` | `/配置` |获取当前内存配置 |
| `获取` | `/配置/提供者` |列出容器中捆绑的 LLM 和嵌入器提供商 |
| `发布` | `/回忆` |创造回忆|
| `获取` | `/回忆` |获取所有内存（按“user_id”、“agent_id”或“run_id”过滤）|
| `获取` | `/memories/{memory_id}` |获取特定记忆 |
| `放置` | `/memories/{memory_id}` |更新记忆 |
| `删除` | `/memories/{memory_id}` |删除特定记忆 |
| `删除` | `/回忆` |删除某个标识符的所有记忆 |
| `获取` | `/memories/{memory_id}/history` |获取内存历史记录 |
| `发布` | `/搜索` |搜索记忆 |
| `发布` | `/重置` |重置所有记忆 |

＃＃＃ 验证|方法|路径|描述 |
|--------|------|-------------|
| `获取` | `/auth/setup-status` |返回“{needsSetup：bool}”。打开，无需授权 |
| `发布` | `/auth/register` |注册第一个管理员。创建第一个管理员后注册将关闭；其他帐户由现有管理员配置。 |
| `发布` | `/auth/login` |交换电子邮件和密码以访问和刷新 JWT |
| `发布` | `/auth/refresh` |将刷新令牌交换为新的访问令牌 |
| `获取` | `/auth/me` |获取当前经过身份验证的用户（需要 JWT） |
| `补丁` | `/auth/me` |更新来电者的姓名或电子邮件。 409 如果新电子邮件已在使用 |
| `发布` | `/auth/更改密码` |更改呼叫者的密码。 401 如果当前密码错误；新密码必须至少为 8 个字符 |

### API 密钥

所有“/api-keys”端点都需要 JWT。|方法|路径|描述 |
|--------|------|-------------|
| `获取` | `/api-keys` |列出调用者的 API 密钥 |
| `发布` | `/api-keys` |创建一个新密钥；完整的 `m0sk_...` 值返回一次 |
| `删除` | `/api-keys/{id}` |撤销 API 密钥 |

### 请求日志|方法|路径|描述 |
|--------|------|-------------|
| `获取` | `/requests?limit=N` |最近的 API 调用日志（JWT 或管理密钥）|

### 实体

|方法|路径|描述 |
|--------|------|-------------|
| `获取` | `/实体` |具有内存计数的不同“user_id”/“agent_id”/“run_id”值 |
| `删除` | `/entities/{entity_type}/{entity_id}` |级联——删除一个实体的所有记忆； `entity_type` 是 `user`、`agent` 或 `run` |

“/auth/*”、“/api-keys”、“/requests”和“/entities”路由对于自托管服务器来说是新的，主要支持仪表板，但您可以直接从自己的工具调用它们。

## 验证该功能是否正常工作

- 点击根路由和“/docs”以确认服务器可访问。
- 运行一个完整的循环：`POST /memories`→`GET /memories/{id}`→`DELETE /memories/{id}`。- 在启动过程中观察服务器日志是否存在导入错误或提供程序配置错误。
- 确认容器重新启动时正确加载环境变量（API 密钥、向量存储凭据）。

---

## 最佳实践1. **保持身份验证开启：** 默认情况下启用身份验证。切勿在生产中设置“AUTH_DISABLED=true”。如果您依赖“ADMIN_API_KEY”，请使用长值（16 个以上字符）或更喜欢每用户 API 密钥。
2. **使用 HTTPS：** 在负载均衡器或反向代理处终止 TLS。
3. **监控正常运行时间：** 跟踪每个端点的请求率、延迟和错误代码。
4. **版本配置：** 将环境文件和 Docker Compose 定义保留在源代码管理中。
5. **限制暴露：** 绑定到专用网络，除非您明确需要公共访问。

使用 Mem0 插件将持久内存添加到 [**OpenAI Codex**](https://openai.com/index/codex/)。 Codex 会忘记任务之间的所有内容 - 该插件通过 MCP 连接到 Mem0 的云存储层并使用基于技能的内存协议自动检索上下文并存储学习内容来解决这个问题。

## 概述

1. **MCP服务器** — 连接Mem0的远程MCP服务器以获取内存工具（添加、搜索、更新、删除）
2. **记忆协议技能** - 指示代理在任务开始时检索记忆，在完成时存储学习内容，并在上下文丢失之前捕获会话状态
3. **插件市场** — 通过 Codex 的存储库级或个人插件市场安装
4. **零本地依赖** — 云托管的 MCP 服务器，无需本地设置

## 先决条件

在使用 Codex 设置 Mem0 之前，请确保您拥有：

1. Mem0 平台帐户和 API 密钥：
   - <a href="https://app.mem0.ai?utm_source=oss&utm_medium=integration-codex" rel="nofollow">在 app.mem0.ai 注册</a>
   - <a href="https://app.mem0.ai/dashboard/api-keys?utm_source=oss&utm_medium=integration-codex" rel="nofollow">获取您的 API 密钥</a>（以 `m0-` 开头）

2. OpenAI Codex 访问3. 在 shell 中导出您的 API 密钥：```bash
export MEM0_API_KEY="m0-your-api-key"
```
## 安装

### 选项 A — 直接 MCP（推荐）

将 Codex 连接到 Mem0 的最快方式 — 无需下载，无需市场。 Codex 从 `~/.codex/config.toml` 读取 MCP 服务器作为 TOML。添加：```toml
[mcp_servers.mem0]
url = "https://mcp.mem0.ai/mcp"
bearer_token_env_var = "MEM0_API_KEY"
```
确保在启动 Codex 的 shell 中导出了“MEM0_API_KEY”，然后重新启动 Codex。

<信息>
  Codex 的“codex mcp add”CLI 仅支持 stdio MCP 服务器。由于 Mem0 的 MCP 是 HTTP/可流式传输，因此您可以通过直接编辑 `config.toml` 来配置它（或通过 Codex 应用程序中的 **插件 → 连接到自定义 MCP → 可流式传输 HTTP** UI）。
</信息>

### 选项 B — 旁加载插件（高级）

为了获得完整的插件体验 - MCP 服务器 **加上** Mem0 SDK 技能、内存协议技能和选择加入生命周期挂钩 - 从本地克隆旁加载插件。 Mem0 存储库已在 [`.agents/plugins/marketplace.json`](https://github.com/mem0ai/mem0/blob/main/.agents/plugins/marketplace.json) 中发布了市场清单，因此无需手动编写 JSON。这遵循 Codex [build-plugins](https://developers.openai.com/codex/plugins/build) 本地测试工作流程。<信息>
  不要将选项 B 与选项 A 结合起来。插件清单通过 [`.codex-mcp.json`](https://github.com/mem0ai/mem0/blob/main/mem0-plugin/.codex-mcp.json) 声明其 MCP 服务器，因此 Codex 在插件加载时自动注册 `mem0` MCP 服务器。将相同的“[mcp_servers.mem0]”块添加到“~/.codex/config.toml”将创建重复注册。
</信息>

**步骤 1.** 将 Mem0 存储库克隆到磁盘上的任意位置：```bash
git clone https://github.com/mem0ai/mem0.git ~/codex-plugins/mem0-source
```
**步骤 2.** 使用 Codex 的 CLI 注册捆绑市场：```bash
codex plugin marketplace add ~/codex-plugins/mem0-source
```
这将 Codex 指向存储库的“.agents/plugins/marketplace.json”。捆绑文件使用 `path: "./mem0-plugin"`，Codex 相对于克隆根解析该路径。

<信息>
  **为什么我们建议这样做而不是手动编写`~/.agents/plugins/marketplace.json`：** Codex 要求任何市场清单中的`source.path` 都是**相对**（以`./` 开头）并且**位于市场根目录内**。存储库的捆绑清单已经满足了这一点 - 市场根目录是克隆目录，并且“mem0-plugin/”位于其中。对于个人“~/.agents/plugins/marketplace.json”，根目录是“~/”，克隆也必须位于“~/”下。 CLI 形式回避了这个限制。
</信息>

**步骤 3.** 重新启动 Codex，运行 `/plugins`，浏览 `Mem0 Plugins` 市场，然后安装 **Mem0**。

**第 4 步（可选）- 启用生命周期挂钩。** Codex 不会从插件清单中自动连接挂钩；它只从“~/.codex/hooks.json”（或“<repo>/.codex/hooks.json”）读取它们。运行一次捆绑安装程序，将 Mem0 条目合并到全局挂钩文件中：```bash
python3 ~/codex-plugins/mem0-source/mem0-plugin/scripts/install_codex_hooks.py
```
然后在`~/.codex/config.toml`中启用hooks功能标志：```toml
[features]
codex_hooks = true
```
重新启动法典。安装程序会注册三个指向克隆内脚本的钩子：

|活动 |行为 |
|--------|----------|
| `会话开始` |加载先前的记忆作为引导上下文 |
| `用户提示提交` |在每次提示之前注入相关记忆 |
| `停止` |提醒代理在回合结束时坚持学习 |

重新运行安装程序是幂等的。要删除挂钩：`python3 ~/codex-plugins/mem0-source/mem0-plugin/scripts/install_codex_hooks.py --uninstall`。

<警告>
  hooks 文件将绝对路径存储到克隆中（例如“~/codex-plugins/mem0-source/mem0-plugin/scripts/...”）。如果您移动或删除克隆，挂钩将悄无声息地中断 - 从新位置重新运行安装程序，或先运行“--uninstall”。
</警告>

### 管理插件

Codex 提供 CLI 命令用于在安装后管理市场：```bash
codex plugin marketplace upgrade               # pull latest plugin versions
codex plugin marketplace remove mem0-plugins   # unregister the marketplace
```
要将更新拉到插件源本身，请在克隆 (`~/codex-plugins/mem0-source`) 中使用 `git pull`，然后运行 ​​`codex 插件市场升级` 来刷新 Codex 的插件缓存。插件缓存在 `~/.codex/plugins/cache/<marketplace>/<plugin>/<version>/` 中。

<信息图标=“检查”>
  在任一选项之后，启动一个新的 Codex 任务并询问：*“列出我的 mem0 实体”* 或 *“搜索我的记忆中的 hello”*。如果“mem0”工具出现并响应，则一切就绪。
</信息>

## 可用的 MCP 工具

安装后，每个 Codex 会话中都可以使用以下工具：|工具|描述 |
|------|-------------|
| `添加内存` |保存用户/代理的文本或对话历史记录 |
| `搜索记忆` |使用过滤器跨记忆进行语义搜索 |
| `获取记忆` |使用过滤器和分页列出记忆 |
| `获取内存` |通过ID检索特定内存|
| `更新内存` |按 ID 覆盖内存文本 |
| `删除内存` |通过ID删除单个内存|
| `删除所有记忆` |批量删除范围内的所有内存|
| `删除实体` |删除用户/代理/应用程序/运行实体及其内存 |
| `列表实体` |列出存储在 Mem0 中的用户/代理/应用程序/运行 |

## 内存协议技巧

当插件被侧面加载时，内存协议技能指示代理：

### 每一个新任务
1. 使用与当前任务相关的查询调用“search_memories”来加载相关上下文
2.回顾返回的记忆以了解之前课程中学到的内容
3. 可以选择调用 `get_memories` 来浏览所有存储的内存

### 完成重要工作后
使用“add_memory”和结构化元数据存储关键知识：|储存什么 |元数据类型 |
|--------------|---------------|
|架构决策 | `{"type": "决定"}` |
|有效的策略| `{"type": "task_learning"}` |
|失败的方法| `{"type": "anti_pattern"}` |
|观察用户偏好 | `{"type": "user_preference"}` |
|环境发现| `{"type": "环境"}` |
|建立的公约| `{"type": "convention"}` |

### 在失去上下文之前
使用元数据“{"type":"session_state"}`存储全面的会话摘要，包括目标、成就、决策、修改的文件和当前状态。

## 故障排除- **“连接失败”** — 验证您的 shell 中是否设置了“MEM0_API_KEY”：“echo $MEM0_API_KEY”
- **没有出现工具** — 安装插件后重新启动 Codex 会话
- **重复 `mem0` MCP 服务器/“工具冲突”错误** — 您将选项 A（直接 MCP）与选项 B（旁载）结合起来。侧面加载的插件会自动从“.codex-mcp.json”注册“mem0”，因此从“~/.codex/config.toml”中删除“[mcp_servers.mem0]”块。
- **`TUI 中插件/读取失败`** — Codex 找不到市场指向的插件目录。如果您使用了“codexpluginmarketplaceadd<path>”，请确认该路径是您的克隆根目录并且“<clone>/.agents/plugins/marketplace.json”存在。如果您手动编写了“~/.agents/plugins/marketplace.json”，则“source.path”必须是相对的（以“./”开头），位于市场根目录内（个人安装为“~/”），并以“mem0-plugin”结尾——例如`”。/codex-plugins/mem0-source/mem0-plugin"`.
- **在`/plugins`中找不到插件** — 再次运行`codex plugin market add ~/path/to/clone`，或确认市场已使用`codex plugin market remove mem0-plugins`注册，然后重新添加。
- **技能未加载** — 验证“plugin.json”中的“skills”字段是否指向包含“SKILL.md”文件的有效目录。
- **Hooks 未触发** — 确认 `codex_hooks = true` 位于 `[features]` 下的 `~/.codex/config.toml` 中，并且 `~/.codex/hooks.json` 包含 Mem0 条目（如果没有，请重新运行安装程序）。启用该标志后重新启动 Codex。
- **移动克隆后挂钩损坏** — 安装程序将绝对路径烘焙到指向克隆内脚本的“~/.codex/hooks.json”中。如果您移动或重命名了克隆目录，请运行“python3 <new-clone>/mem0-plugin/scripts/install_codex_hooks”。py` 从新位置 - 安装程序是幂等的并替换旧条目。<卡组列={2}>

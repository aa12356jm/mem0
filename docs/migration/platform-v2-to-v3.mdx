## 概述

新的 Mem0 内存算法对内存的提取、存储和检索方式进行了彻底的重新设计。它在 LoCoMo** 上得分 **91.6，在 LongMemEval** 上得分 **93.4，比之前的算法分别提高了 20 和 26 分，同时将提取延迟大约减少了一半。|发生了什么变化|之前 |之后 |
|---|---|---|
| **提取** |两次LLM通过（提取+合并）|单次仅 ADD（一次 LLM 调用）|
| **记忆突变** |添加、更新、删除|仅添加 — 不会覆盖或删除任何内容 |
| **代理生成的事实** |经常被忽视|一流，同等重量存放|
| **实体链接** |不可用 |跨记忆提取和链接实体|
| **图形存储器** |独立的图形存储+仪表板可视化|被内置实体链接取代，平台仪表板上没有图形视觉效果 |
| **检索** |仅语义（矢量）|结合多个信号的混合检索 |## 这对您的应用意味着什么

### 记忆是积累而不是被覆盖

以前的算法可以在提取过程中更新或删除现有的内存。新算法仅添加新事实。当信息发生变化时（例如，用户从纽约搬到旧金山），这两个事实都会在时间上下文中保留。这意味着：

- **随着时间的推移，更多的记忆** - 你的记忆数量将会增长而不是停滞不前
- **更好的时间推理** - 系统可以区分“曾经住在纽约”和“现在住在旧金山”
- **没有信息丢失** - 保留了看似矛盾但实际上互补的事实

<提示>
如果您的应用程序以前依赖更新/删除行为来保持较低的内存计数，则新算法会在检索时处理此问题。多信号检索将最相关的当前信息排名靠前，而不会破坏历史背景。
</提示>

### 代理生成的事实现在已被捕获以前，当客服人员说“我已经预订了您 3 月 3 日的航班”之类的话时，系统通常会忽略它，只存储用户明确指出的内容。新算法将代理生成的事实视为一流的记忆。如果您的应用程序涉及确认操作、提供建议或共享信息的代理，您将发现对这些交互的回忆明显更好。

### 现在检索是混合的

搜索现在使用混合检索，这提高了排名质量 - 特别是对于涉及精确关键字、专有名词或出现在多个内存中的实体的查询。响应形状不变：```json
{
  "results": [
    {
      "id": "mem-uuid",
      "memory": "User moved to San Francisco in January 2026",
      "score": 0.82,
      "metadata": {},
      "categories": ["location"]
    }
  ]
}
```
顶级“score”仍然是“[0, 1]”值。结果之间的相对排名与 v2 相当，但由于评分方法发生变化，绝对数字发生变化 - 针对代表性查询重新调整应用程序中的任何硬阈值。

## API 变更

### 新的 V3 端点

新算法可通过 V3 API 获取。端点分为每个操作的路径：

|运营| SDK方法|端点|
|---|---|---|
|添加回忆 | `client.add()` | `POST /v3/memories/add/` |
|搜索记忆 | `client.search()` | `POST /v3/memories/search/` |
|获取所有记忆（分页）| `client.get_all()` | `POST /v3/memories/` |

<信息>
`get_all` / list 现在返回一个分页信封：`{"count": int, "next": str | null，“上一个”：str |空，“结果”：[...]}`。将 `page` 和 `page_size` 作为查询参数传递以进行分页；默认返回第一页。
</信息>

<代码组>```python Python

## SDK 重大变更

除了算法更新之外，Python 和 TypeScript 客户端 SDK 也已清理。这些更改会影响您初始化和调用客户端的方式。

### Python 客户端 SDK```python
from mem0 import MemoryClient

# Before
client = MemoryClient(
    api_key="...",
    org_id="org-1",       # [REMOVED] Removed
    project_id="proj-1"   # [REMOVED] Removed
)
client.add(messages, user_id="alice", async_mode=True, output_format="v1.1")

# After
client = MemoryClient(api_key="...")
client.add(messages, user_id="alice")
# async_mode and output_format removed (async by default, v1.1 always)
```
**删除的参数：** `org_id`、`project_id`、`api_version`、`output_format`、`async_mode`、`enable_graph`、`immutable`、`expiration_date`、`filter_memories`、`batch_size`、`force_add_only`、`includes`、`excludes`、`keyword_search`、`org_name`、`project_name`

### TypeScript 客户端 SDK

所有参数现在都使用 **camelCase** （SDK 自动处理与 API 之间的转换）：```typescript
// Before
const client = new MemoryClient({
  apiKey: "...",
  organizationId: "org-1",  // [REMOVED] Removed
  projectId: "proj-1"       // [REMOVED] Removed
});
await client.search("query", {
  user_id: "alice",          // [REMOVED] snake_case
  top_k: 20,                 // [REMOVED] snake_case
  enable_graph: true          // [REMOVED] Removed
});

// After
const client = new MemoryClient({ apiKey: "..." });
await client.search("query", {
  filters: { userId: "alice" },  // [OK] inside filters
  topK: 20                       // [OK] camelCase
});
```
**删除：** `OutputFormat` 枚举、`API_VERSION` 枚举、`organizationId`、`projectId`、`organizationName`、`projectName`、`enableGraph`、`asyncMode`、`outputFormat`、`immutable`、`expirationDate`、`filterMemories`、`batchSize`、`forceAddOnly`、`includes`、`exclusions`、`keywordSearch`

<信息>
有关所有 SDK 的参数更改的完整列表，请参阅 [OSS 迁移指南](/migration/oss-v2-to-v3#removed-parameters-reference)。

## 图存储器→实体链接

图内存已被**内置实体链接**取代。变化：- **图形可视化从平台仪表板中删除。**项目仪表板中的图形视图不再可用。
- **`enable_graph` 项目设置已删除。** 开关已从仪表板中消失； API 参数被忽略。
- **无需配置外部图形存储。** 以前的图形内存需要单独的 Neo4j（或类似）部署。实体链接在平台内部本地运行——无需配置，无需管理连接字符串。
- **实体链接是本机替换。** 实体（专有名词、引用文本、复合名词短语）会自动从每个记忆中提取，并在属于同一用户的记忆之间进行链接。在搜索时，查询中的实体与该索引进行匹配并用于提高排名。提升值会合并到每个结果返回的组合“分数”中。

**无需迁移工作。** 对于新算法上的所有项目，实体链接都会自动激活。现有的记忆不会被重新处理，但您添加的任何新记忆都将被索引，以便将来进行基于实体的检索。<注意>
如果您的应用程序之前从 API 响应中读取图形关系（搜索结果中的“关系”字段），请注意该字段不再填充。实体关系现在通过检索排名间接使用，而不是作为单独的图形结构公开。
</注>

## 迁移清单<步骤>
  <Step title="查看您的搜索阈值">
    默认的“阈值”现在为“0.1”（以前没有阈值）。如果您的应用程序依赖于未过滤的结果，请在搜索调用中显式传递“threshold=0.0”以保留旧行为。在大多数情况下，新的默认值更好——它过滤掉了低相关性的噪音。
  </步骤>
  <Step title="查看重新排名使用情况">
    重新排名现在默认为“false”。如果您的应用程序依赖于重新排名的结果，请将“rerank=True”添加到您的搜索调用中。请注意，重新排名会增加延迟（约 200-400 毫秒），但可以提高复杂查询的排序质量。
  </步骤>
  <Step title="更新分数处理（可选）">
    顶级“score”字段继续像以前一样工作。现在它是组合的多信号分数（语义+关键字+实体）而不是纯粹的余弦相似度，因此绝对数字会有所不同。相对排名仍然具有可比性 - 如果您的应用程序中有基于阈值的过滤，请在代表性查询集上重新调整。
  </步骤>
  <Step title="调整内存计数预期">
    通过仅添加提取，内存计数将随着时间的推移而增长，而不是被整合。这是设计使然——检索处理相关性排名。如果对内存计数有硬性限制，请考虑使用内存过期功能或定期清理。
  </步骤>
  <Step title="使用代表性查询进行测试">
    最大的改进是时间推理（LoCoMo 上+29.6）、多跳查询（+23.1）和辅助记忆回忆（LongMemEval 上+53.6）。测试这些类别中的查询以查看改进情况。
  </步骤>
</步骤>## 向后兼容性

- **V1 和 V2 端点继续工作。** 不需要立即迁移到 V3 端点。
- **保留现有记忆。**新算法不会修改或重新处理以前存储的记忆。
- **搜索响应形状不变。** 顶级 `score` 和 `results[]` 数组相同；读取“score”的现有代码继续有效。改变的是数字背后的评分方法（多信号融合而不是纯余弦），因此即使排名保持可比，绝对值也会发生变化。
- **列表响应形状已更改。** `get_all` 现在返回分页信封 (`{count, next, previous, results}`)，而不是裸露的 `{results: [...]}`。更新读取 `response["results"]` 的代码以继续工作，或切换到处理这两种形状的客户端 SDK。## 性能改进

|公制|以前的算法 |新算法 |
|---|---|---|
| **LoCoMo 整体服** | 71.4 | 71.4 **91.6** (+20.2) |
| **LongMemEval 总体** | 67.8 | 67.8 **93.4** (+25.6) |
| **提取延迟 (p50)** | ~2.0 秒 | **~1.0s** |
| **每个查询的平均标记** | — | 6.8-7.0K（前200名）|所有基准测试都在代表生产的堆栈上运行 - 故意避免使用前沿模型，以保持数字代表实际生产工作负载。

＃＃ 常问问题<手风琴组>
  <Accordion title="我需要重新处理我现有的记忆吗？">
    不会。现有的记忆保持原样。推出后添加的新内存将使用新的提取算法。新旧记忆都可以通过相同的检索管道进行搜索。
  </手风琴>
  <Accordion title="我的内存计数现在会增加得更快吗？">
    是的。仅 ADD 方法意味着记忆积累而不是巩固。这是有意为之的——检索系统处理排名和相关性。如果您需要管理内存量，请使用过期日期功能或删除 API。
  </手风琴>
  <Accordion title="我可以选择退出新算法吗？">
    新算法是所有平台用户的默认算法。如果您有特定需要使用以前的提取行为，请联系支持人员。</手风琴>
  <Accordion title="实体链接如何影响我现有的集成？">
    实体链接是自动且透明的。它可以提高检索质量，而无需对集成进行任何更改。实体从新的记忆和搜索查询中提取，并自动匹配。
  </手风琴>

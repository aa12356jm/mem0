## 概述

新的 Mem0 版本重新设计了提取和检索，并清理了 Python 和 TypeScript 的 SDK 表面：

- **提取**：单次仅添加（一次 LLM 调用，无更新/删除）
- **检索**：多信号混合搜索（语义+BM25关键词+实体匹配）
- **实体链接**：自动实体提取和跨内存链接
- **SDK清理**：删除不推荐使用的参数，标准化命名约定
- **API 表面与平台一致**：实体 ID 现在在 OSS 和平台上遵循相同的约定 - 顶级 kwargs 用于“add()”/“delete_all()”，在“filters”内用于“search()”/“get_all()”

这些变化使 LoCoMo** (71.4 → 91.6) 提高了 **+20 点，对 LongMemEval** (67.8 → 93.4) 提高了 **+26 点，同时将提取延迟大约减少了一半。

### 6.实体商店设置

新算法会自动创建一个名为“{your_collection}_entities”的并行实体存储集合。无需手动设置——它是在首次使用时创建的。

## 图存储器→实体链接

开源 SDK 中已删除图形存储支持。它被**内置实体链接**取代，后者在本机运行，没有外部依赖项。**删除了什么：**
- `enable_graph` / `enableGraph` 配置标志
- `graph_store` / `graphStore` 配置块（Neo4j、Memgraph、Kuzu、Apache AGE、Neptune）
- 所有图形内存代码路径（约 4000 行）

**取代它的是：**实体链接在添加管道期间从每个内存中提取实体（专有名词、引用文本、复合名词短语），并将它们存储在现有向量存储内的并行集合（`{collection}_entities`）中。在搜索时，查询中的实体与该集合进行匹配，并用于增强相关记忆。提升将合并到每个结果的组合“分数”中。

**迁移：**
- 从配置中删除 `enable_graph` / `enableGraph`
- 删除 `graph_store` / `graphStore` 块——它不再被读取
- 如果您仅将图形驱动程序（neo4j、memgraph 等）用于 Mem0，请卸载它们
- 无需数据迁移。实体链接在下一个“add()”调用时自动激活。

<警告>
不再填充通过搜索结果中旧的“关系”字段公开的图形关系。实体关系通过检索排名间接使用，而不是作为可查询的图形结构公开。如果您的应用程序直接依赖于遍历图形关系，您将需要根据新的 API 重新设计该部分。
</警告>

## 新算法如何工作

### 提取：单通道 ADD-Only```
Input conversation
    → Retrieve top-10 related existing memories (for deduplication context)
    → Single LLM call: extract all distinct new facts
    → Batch embed extracted memories
    → Hash-based deduplication (MD5, prevents exact duplicates)
    → Batch insert into vector store
    → Entity extraction + linking
```
之前的算法使用了两个 LLM 调用——一个用于提取候选事实，一个用于决定对现有内存的添加/更新/删除操作。新算法将其折叠成一个仅进行相加的调用。该模型将其能力用于理解输入，而不是与现有状态进行比较。

### 检索：多信号混合搜索```
Query
    → Preprocess (lemmatize keywords, extract entities)
    → Parallel scoring:
        1. Semantic search (vector similarity)
        2. BM25 keyword search (normalized term matching)
        3. Entity matching (entity graph boost)
    → Score fusion → Top-K selection
```
**评分：** 将三个信号标准化并融合为每个结果的单个组合“分数”。融合根据运行时可用的信号进行调整（仅语义、语义 + BM25，或当 spaCy + 实体存储处于活动状态时全部三种）。

**BM25 是一个提升信号，而不是召回扩展器。** 只有语义搜索结果才是候选者 - BM25 和实体分数可以提升排名，但不会添加新的候选者。## 矢量存储兼容性

所有 15 个受支持的矢量存储都已通过两项新功能得到增强：

|能力|目的|如果不受支持则回退 |
|---|---|---|
| `keyword_search()` | BM25/全文关键词匹配 |退回到纯语义搜索 |
| `search_batch()` |批量搜索实体匹配 |退回到顺序搜索 |

**特定于 Qdrant 的更改：**
- 现在在同一集合中使用稀疏向量 (BM25) 和密集向量
- 需要用于 BM25 编码的“fastembed”库（延迟加载，优雅降级）
- 安装：`pip install fastembed`

**所有其他矢量商店：**
- 使用原生全文功能通过“keyword_search()”方法进行增强
- 不需要额外的依赖

```
### TypeScript 客户端：snake_case 参数不再起作用

所有 TypeScript 客户端 SDK 参数现在都使用驼峰命名法。 SDK 自动处理与 API 之间的转换：```typescript
// Before
await client.search("query", { user_id: "alice", top_k: 20 });

// After
await client.search("query", { filters: { userId: "alice" }, topK: 20 });
```
### `ValueError：search() / get_all() 不支持顶级实体参数`

`search()` 和 `get_all()` 现在需要 `filter` 内的实体 ID。顶级 kwargs 引发 `ValueError`。这使 OSS SDK 与平台 API 保持一致。```python
# Before
results = m.search("query", user_id="alice", top_k=20)

# After
results = m.search("query", filters={"user_id": "alice"}, top_k=20)
```
`add()` 和 `delete_all()` 继续接受实体 ID 作为顶级 kwargs。

### 搜索返回的结果比以前少

默认“阈值”从“无”更改为“0.1”。以前包含的低相关性结果现在已被过滤掉。要恢复旧行为：```python
results = m.search("query", filters={"user_id": "alice"}, threshold=0.0)
```
### 未找到 spaCy 模型

如果您看到有关缺少 spaCy 模型的错误，请下载所需的模型：```bash
python -m spacy download en_core_web_sm
```
如果根本没有安装 spaCy，请安装 NLP 附加组件：```bash
pip install "mem0ai[nlp]"
```
### 实体存储集合创建失败

实体存储尝试自动创建“{collection_name}_entities”集合。如果您的矢量数据库具有受限权限，请使用与主集合相同的嵌入尺寸预先创建此集合。

### 分数值与之前不同

顶级 `score` 的范围仍然是 `[0, 1]`，但在 v3 中计算方式有所不同。结果之间的相对排名保持可比，但绝对数字发生变化 - 根据代表性查询重新调整应用程序中的任何硬阈值。

如果您需要特定用例的原始余弦相似度，请通过“vector_store.search(...)”直接针对矢量存储运行未增强的矢量查询。
